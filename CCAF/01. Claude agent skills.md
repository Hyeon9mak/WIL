## What are skills? Where they live?
```
--- 
name: frontend-security-accessibility-reviewer
description: "Use this agent when you need to review frontend code for accessibility..." 
---
```

- skill 은 Claude Code 가 알아서 특정 작업을 처리하는 방법을 알려주는 마크다운 명세다.
	- 똑같은 기술 명세를 매 번 알려주는 대신, 한 번만 작성해두면 비슷한 작업 때마다 Claude Code 가 이를 참고하고 적용한다.
- `.claude/skills` 폴더는 Claude Code 가 읽을 수 있는 skill 들을 담아놓는 폴더다.
	- 그 하위에 폴더명으로 각 동작을 명세해둔다.
- `.claude/skills/{skill-name}/SKILL.md` 에 내용을 명세한다.
	- `name` 은 skill 을 구분 짓는 방식
	- `description` 은 Claude 가 그 skill 을 사용할지 말지 결정하는 기준이 된다.
	- 사용자의 요청을 읽고, 사용 가능한 모든 skill 설명과 비교한 뒤 일치하는 skill 을 활성화 시킨다.
- 개인 단위 skill 을 모아두고 싶다면 `~/.claude/skills` 에 모아두면 된다.
	- 모든 프로젝트에서 참고 된다.
- 프로젝트 단위 skill 을 격리하고 싶다면 `.claude/skills` 에 모아두면 된다.

## Skills / CLAUDE.md / subagents / Hooks / MCP / Slash Commands
skills 을 CLAUDE.md, subagents, hooks, and MCP servers 들과 구별해보자.
이들의 사용처를 구별할 줄 알면, 잘못된 활용을 방지할 수 있다.

### skill
- 내 요청에 걸맞는 동작이 자동으로 적용 되길 기대할 때.
- **Claude Code 가 내부적으로 `name`, `description` 만 읽기 때문에 context 를 더 넓게 확보할 수 있다!!**
### CLAUDE.md
- 모든 대화에 로드된다. 엄격한 기준을 원한다면 그것은 CLAUDE.md 에 기입해야한다.
	- 예를 들면 "DB schema 절대 변경 금지" 와 같은 규칙
- 반면 skill 은 '필요에 의할 때' 만 로드된다. 반대로 말하면 불필요할 땐 context 낭비 방지.
	- 그 외 context 를 더럽힐 수 있는 모든 정보를 skill 로 분류하면 좋다.
	- "아주 상세한 절차들" 등
### subagents
- subagents 들은 agent 로부터 완전히 분리된 개별 context 를 구성해서 요청사항을 전달 받고, 작업을 수행한 후 결과만을 반환한다.
- 이 과정을 통해 subagent 가 작업 중에 context 를 훼손하는 것으로부터 agent 가 자유로워진다.
- 즉, 하위 속성의 작업을 context 분리 후 수행 시키는 것이 핵심
- 반면 skill 은 현재 context 에 별개 정보를 더해주는 것이다.
	- "이 상세한 내용들이 대화 전반에 사용되는 경우"
### Hooks
- 이벤트 기반 동작. 파일 저장 시, tool 호출 시, 무언가 수행 전/후 로.
	- 사이드 이펙트와 같다고 보면 된다.
	- "모든 파일 저장에서 실행되어야 하는 연산들", "작업을 마무리해서 branch 에 push 하면, PR 을 생성한다." 와 같은 hook 을 설정해볼 수 있겠다.
- 반면 skill 은 사용자의 요청을 description 과 매칭시켜 해석한 것 기반(요청하는 내용에 따라 활성화)
	- Claude 의 추론에 영향을 미치는 내용들을 skill 로 지정.
### MCP
- 외부 MCP 들과 통신에 필요한 정보
- context 나 이런 것과는 완전히 관련이 없는 다른 정보다.
### Slash Commands
- 개발자가 알아서 직접 입력하는 것.
- `CLAUDE.md` 에 Slash Commands 사용을 명세할 순 있으나, 이를 꼭 활용하진 않음. 확률적이다.

### Skill vs Slash Command 의 트리거 방식 차이
같은 동작이라도 어디에 두느냐에 따라 발동 방식이 완전히 다르다.

- **Slash command**: 사용자가 `/pull-request:create` 같은 식으로 **명시 호출**해야만 발동. 자동 트리거 없음. "PR 만들어줘" 같은 자연어로는 동작 X.
- **Skill**: `description` 매칭으로 **자동 발동**. "PR 만들어줘", "리뷰 부탁" 같은 자연어 요청에 Claude 가 알아서 매칭해서 활성화한다.

판단 기준은 단순함. `/` 를 자주 친다면 slash command 가 명시적이라 편하고, `/` 를 거의 안 친다면 skill 로 전환하는 게 합리적이다. 자연어로만 부탁하면서 slash command 로만 정의해두면 Claude 가 그 기능의 존재 자체를 모름.

중요한 것은 이것들에 목매여서 하나로 통일 시킬 필요는 없다. 다 일련의 tool 들일 뿐이니 조화롭게 사용하는 것이 가장 좋음.

## When to use Skills
특정 업무에 적용되는 전문 지식에 가장 적합하다.

- 팀이 따르는 코드 검토 기준
- 선호하는 commit message 형식
- 브랜드 가이드라인, 특정 유형의 문서 템플릿
- 디버깅 체크 리스트

더 간단하게 요약하면, Claude 에게 (케이스마다 선택적으로) 같은 내용을 반복해서 설명하는 일이 잦다면 그건 skill 로 뭉쳐둬야 한다.

## 동작 매커니즘
skill 생성 이후 세션을 다시 시작하는 것은 필수.

### 권한 검사 흐름
skill 의 tool 호출은 **두 레이어 AND 조건**으로 검사된다. 둘 다 통과해야 실행됨.

1. **SKILL.md frontmatter 의 `allowed-tools`** — skill 이 활성화됐을 때 사용 가능한 tool *종류* 화이트리스트. skill 작성자의 자기 제약.
2. **`.claude/settings.local.json` 의 `permissions.allow/ask/deny`** — tool 호출의 *내용*을 패턴으로 검사. 사용자 환경 단위 마찰 조절.

검사 순서:
```
tool 호출
  ↓
[1] allowed-tools 화이트리스트 검사
  ├ 없음 → 조용히 차단 (prompt 없음)
  └ 통과 ↓
[2] settings.local.json 패턴 매칭
  ├ deny 매치  → 차단
  ├ allow 매치 → 즉시 실행
  ├ ask 매치   → 사용자 confirm prompt
  └ 매치 없음  → confirm prompt (default ask)
```

케이스별:
- **A.** `allowed-tools` 에 없는 tool 호출 → 조용히 차단. 사용자한테 prompt 도 안 뜸. skill 작성자가 의도적으로 제한한 경계.
- **B.** `allowed-tools` 통과 + settings 에 매치 없음 → confirm prompt. 사용자가 매번 확인.
- **C.** 둘 다 통과 (`allowed-tools` 안에 있고 settings 에 `allow` 매치) → 즉시 실행, 마찰 0.

즉 `allowed-tools` 는 *능력의 상한*이고 settings 는 *그 안에서의 마찰 조절*이다.

만약 이름이 겹친다면, 다음과 같은 우선 순위로 skill 들을 탐색하게 된다.
1. Enterprise: `managed-settings.json`
2. Personal: `~/.claude/skills`
3. Project: `project/.claude/skills`
4. Plugin: `project/.claude-plugin/plugin.json`
	1. `project/.claude-plugins/skills`

조직 차원에서 skill 을 강제하고 싶다면 Enterprise 레벨을 이용하자. 그리고 가능하면 더 자세하고 직관적인 skill 명칭을 작성하여 충돌을 회피하는 것이 가장 좋겠다.

skill 을 업데이트 하고 싶다면 `SKILL.md` 파일 내부를 편집하면 된다!
skill 을 제거하고 싶다면 디렉토리를 제거하자.
당연히 이후 세션을 다시 시작하는 것은 필수.

## Additional Metadata Field
- `name` 과 `description` 말고도 강력한 추가 metadata field 들을 기입할 수 있다.
- [https://agentskills.io/specification](https://agentskills.io/specification) 참고
- 중요 사항들은 아래와 같다.
	- `name`: 필수. skill 식별에 활용. 소문자, 숫자, 하이픈만 사용 가능. 최대 64자이며 디렉토리 명칭과 일치해야 한다.
	- `description`: 필수. Claude 가 skill 판별에 활용. 최대 1024 자.
	- `allowed-tools`: skill 이 활성화된 상태에서 Claude 가 사용할 수 있는 tool 을 제한.
	- `model`: 해당 skill 에 사용할 Claude model 을 직접 지정

### about description
당연히 그렇듯, skill 이 제대로 동작하지 않는다면 (호출되지 않는다면) `description` 이 부족한 것이다. 관련된 키워드를 더 많이 넣어서 상세히 구술하는 것이 좋다.

### about allowed-tools
보안이 중요하다면 읽기만 허용하고, 쓰기를 막아야할 수도 있다. 이를 가능하게 하는 것이 `allowed-tools`.
명세된 것 외에 다른 기능은 하지 않는다. (`Read, Grep, Glob, Bash` 등)

`settings.local.json` 과 어떻게 맞물리는지는 위 `### 권한 검사 흐름` 참고. `allowed-tools` 는 능력의 상한, settings 는 마찰 조절이라는 역할 구분이 핵심.

## skill 을 가볍게 유지하기
Claude 는 매칭되는 skill 을 획득했을 때, skill 의 상세 내용들을 context 로 불러들인다. 다만 때때로 해당 내용들을 소화하기 위해 별도 참고자료, 예시, 스크립트가 필요할 수 있는데. 이걸 모두 SKILL.md 에 적어두면 context 가 지나치게 방대해진다. (유지보수하기도 빡셈)

이 문제를 쉽게 해결하는 것이 Progressive Disclosure (점진적 공개)
필수 지침만 `SKILL.md` 에 보관하고, 자세한 참고 자료는 별도 파일에 보관하고, 참조만 넣어주면 된다.

표준은 다음과 같은 정리방식을 권장한다.

- **scripts/** — 실행 가능한 코드
- **references/** —  추가 문서, 참고 문헌
- **assets/** — Images, templates, or other data files  

좋은 법칙은 `SKILL.md` 을 500줄 이하로 유지하는 것이다. 그 이상이라면 콘텐츠를 분리해야한다는 신호.

실제 예시. `pr-create-or-update` skill 의 디렉터리 구조:
```
.claude/skills/pr-create-or-update/
├── SKILL.md                  # 본체. 흐름·판단 기준만.
├── scripts/
│   ├── resolve-pr.sh         # 현재 브랜치 → 기존 PR 번호 조회 (read-only)
│   ├── fetch-meta-and-diff.sh# Jira 키·커밋·diff 수집 (read-only)
│   ├── create-pr.sh          # gh pr create (write)
│   └── update-pr.sh          # gh pr edit (write)
├── references/
│   └── pr-body-template.md   # PR 본문 골격
└── assets/
    └── labels.json           # 라벨 매핑 등 데이터
```
`SKILL.md` 본문에는 "scripts/resolve-pr.sh 를 실행해 현재 PR 을 찾는다" 정도만 적힘. 스크립트 본문은 context 에 안 들어옴.

## scripts 를 효과적으로 사용하기
사실 skill 디렉터리의 script 들은 그 내용을 context 에 불러들이지 않고도 실행시킬 수 있다.
context 에 읽어오는 것은 출력 token 만 소비하는 것.

우리가 집중해야하는 것은 Claude Code 에게 실행을 요청하는 것이지, 읽는 걸 요청하는게 아니다.

- Environment validation
  (환경 검증)
- Data transformations that need to be consistent
  (일관성이 필요한 데이터 변환)
- Operations that are more reliable as tested code than generated code
  (생성된 코드보다 테스트된 코드로서 더 신뢰할 수 있는 연산들)

### Before / After 비교

**Before** — SKILL.md 본문에 셸 명령 나열:
```markdown
1. `git rev-parse --abbrev-ref HEAD` 로 현재 브랜치 확인
2. `git log origin/develop..HEAD --oneline` 으로 커밋 수집
3. `gh pr list --head $BRANCH --json number` 로 기존 PR 조회
4. 없으면 `gh pr create --base develop --title "..." --body "..."`
```
Claude 가 한 줄씩 Bash 호출. 인용·변수 치환 실수 가능. 매 호출마다 권한 검사. settings 에 `Bash(gh pr create:*)`, `Bash(gh pr list:*)`, `Bash(git log origin/develop..HEAD:*)` 등이 N 줄씩 누적된다.

**After** — `scripts/create-pr.sh` 한 곳에 넣음:
```bash
#!/usr/bin/env bash
set -euo pipefail
BRANCH=$(git rev-parse --abbrev-ref HEAD)
COMMITS=$(git log origin/develop..HEAD --oneline)
# ... gh pr create ...
```
SKILL.md 에서는 한 줄.
```markdown
3. `scripts/create-pr.sh "$JIRA_KEY" "$TITLE"` 실행
```
Claude 는 `Bash(scripts/create-pr.sh ...)` 한 번만 호출. **스크립트 내부 명령은 권한 검사 대상이 아니다** — Claude Code 는 최상위 Bash 호출만 검사함.

### 권한 단순화 효과

settings.local.json **Before**:
```json
{
  "permissions": {
    "allow": [
      "Bash(git rev-parse:*)",
      "Bash(git log origin/develop..HEAD:*)",
      "Bash(git diff origin/develop..HEAD:*)",
      "Bash(gh pr list:*)",
      "Bash(gh pr view:*)"
    ],
    "ask": [
      "Bash(gh pr create:*)",
      "Bash(gh pr edit:*)",
      "Bash(gh pr comment:*)"
    ]
  }
}
```
**After**:
```json
{
  "permissions": {
    "allow": [
      "Bash(.claude/skills/pr-create-or-update/scripts/resolve-pr.sh:*)",
      "Bash(.claude/skills/pr-create-or-update/scripts/fetch-meta-and-diff.sh:*)"
    ],
    "ask": [
      "Bash(.claude/skills/pr-create-or-update/scripts/create-pr.sh:*)",
      "Bash(.claude/skills/pr-create-or-update/scripts/update-pr.sh:*)"
    ]
  }
}
```
스크립트 경로 단위로 묶이니 권한 정책 = 의도 단위가 된다. "PR 생성 동의" 한 줄.

### 부수 효과
- Claude 의 인용·이스케이프 실수 회피 (셸 인자 처리는 스크립트 안에서 한 번에).
- 결정론적 동작. 같은 입력 → 같은 결과.
- context 절약. 스크립트 본문은 SKILL.md 가 호출만 시키지, context 에 안 올라옴.
- 사람이 직접 실행해서 디버깅 가능.

### 분리하면 안 되는 부분
모든 걸 스크립트로 빼면 안 됨. SKILL.md 에 남겨야 하는 것:
- **MCP 호출** — `mcp__atlassian__jira_get_issue` 같은 호출은 Claude 가 직접 해야 함. 셸에서 못 부름.
- **사용자 confirm 흐름** — "이 제목으로 PR 생성할까요?" 같은 대화형 분기는 Claude 가 판단.
- **PR 본문 생성** — commit message · diff 보고 본문을 *작성*하는 일은 LLM 의 영역. 스크립트는 데이터 수집만, 글쓰기는 Claude.
- **분기 판단** — 변경 크기 보고 메인 vs subagent 결정 같은 메타 판단.

원칙: **결정론적인 부분 → 스크립트, LLM 판단 필요 → SKILL.md.**

## 읽기 권한 vs 쓰기 권한 분리
스크립트 단위로 권한을 묶을 때 자연스럽게 따라오는 패턴. 부작용 없는 것과 외부에 흔적 남는 것을 다르게 취급한다.

- **read-only** (`resolve-pr.sh`, `fetch-meta-and-diff.sh`) → `allow`. 매번 통과. 마찰 0.
- **write / 외부 가시** (`create-pr.sh`, `update-pr.sh`, `post-comment.sh`) → `ask`. 매번 사용자 confirm.

```json
{
  "permissions": {
    "allow": [
      "Bash(.claude/skills/pr-create-or-update/scripts/resolve-pr.sh:*)",
      "Bash(.claude/skills/pr-create-or-update/scripts/fetch-meta-and-diff.sh:*)",
      "Bash(.claude/skills/pr-review/scripts/fetch-diff.sh:*)"
    ],
    "ask": [
      "Bash(.claude/skills/pr-create-or-update/scripts/create-pr.sh:*)",
      "Bash(.claude/skills/pr-create-or-update/scripts/update-pr.sh:*)",
      "Bash(.claude/skills/pr-review/scripts/post-comment.sh:*)",
      "Bash(.claude/skills/pr-transition/scripts/transition-issue.sh:*)"
    ]
  }
}
```

이 분리는 "destructive / external 동작은 confirm 받자" 라는 개인 원칙과 그대로 맞닿는다. 읽기는 자동, 쓰기는 손가락 한 번 — 보안과 마찰의 균형점.

## Skill 공유
- 모든 지식이 그렇겠지만, Claude skill 은 팀원들과 함께 공유하며 사용하면 더욱 효과적이다.
- 모두가 같은 skill 을 사용하면 더 일관성 높고 뛰어난 품질의 효과를 기대할 수 있다.
### Git 을 이용한 공유
- 현재 가장 간단한 공유 방식은 Git 에 commit 하는 것.
	- 자연스럽게 버전 관리 역시 진행되며 별도 설치 절차도 없다.
### Plugin
- 또 다른 방법은 Claude Plugin 을 이용하는 것
- plugin 프로젝트 내부에 기존 skill 과 같이 skill/{skill-name}/SKILL.md 를 구성하고, 이를 마켓 플레이스에 배포하면 다른 사용자가 다운로드 받아 그대로 사용할 수 있다.
- Git 을 이용한 공유처럼 특정 프로젝트에만 국한되지 않고, 필요한 모든 사람이 직접 받아서 활용할 수 있는 방식.
### Enterprise deployment
- 관리자는 직접 모든 구성원들에게 SKILL 을 배포할 수 있다.
	- skill 해석 우선 순위에서 enterprise 가 가장 높은 우선 순위를 갖는 점을 이용하는 방식.
- 조직 전체에서 유지해야하는 필수 표준, 보안 요구사항, 코드 관행 등을 지키기 매우 좋다.
- 핵심 키워드는 `must`. `must` 를 포함시키자.
### Sharing to subagents
- **subagent 들은 agent 의 skill 내용을 참조하지 못한다!!!!!!!**
- subagent 들은 컨셉 자체가 "완전히 새로운 context 에서 작업 수행" 이므로, context 에 새로운 지식을 추가하는 skill 과 친하지 않은 것.
- custom agent (`.claude/agents/`)의 frontmatter `skills` field 를 채우지 않는 한 skill 을 참조하지 못한다.
	- 즉 인스턴트하게 생성된 subagent 들은 skill 을 참조할 방법이 없다.
	- custom agent 들은 세션이 시작될 때 skill 을 바로 다 불러온다. main agent 처럼 대화형이 아님.
- BE code review subagent 와 FE code review subagent 에게 각각 필요한 skill 들이 다를 것이다.
	- 이 때문에 agent 역시도 세분화를 많이 해두는 것이 당연히 좋다.
```
--- 
name: frontend-security-accessibility-reviewer
description: "Use this agent when you need to review frontend code for accessibility..." 
tools: Bash, Glob, Grep, Read, WebFetch, WebSearch, Skill... 
model: sonnet 
color: blue 
skills: accessibility-audit, performance-check
---
```

## Skill 이 제대로 동작하지 않을 때
- skill 이 동작하지 않거나
- 로드되지 않거나
- 충돌이 발생하거나
- 런타임 오류가 발생하거나
이런 상황들을 타개할 방법이 생각보다 쉽다.

### Agent Skills Verifier
- `skills-ref` 명령어를 통해 검증하는 것.
	- 운영체제에 따라 다르겠지만 주로 `uv` 명령어를 통해 설치하면 됨
### Skills aren't triggering
- 원인은 대부분 `description` 이 명확하지 않아서 매칭되지 않는 것.
- `description` 을 더 자세히 작성하고 키워드를 많이 부여하는게 정답이다.
	- 사용자가 실제로 타이핑 할 만한 트리거 문구를 넣는 것.

### Skills Doesn't Load
- 파일 경로, 대소문자 문제.
- `claude --debug` 명령어를 실행하면 로딩 오류를 체크할 수 있다.

### Wrong Skill Gets Used
- 당연히, `description` 이 너무 비슷해서 혼동하는 것.
- 구분을 더 명확하게 하는 수 밖에. 더 명확한 사용 사례를 나누는 것.
	- 그 방향이 자기 자신한테도 도움이 된다.

### Skill Priority Conflicts
1. Enterprise: `managed-settings.json`
2. Personal: `~/.claude/skills`
3. Project: `project/.claude/skills`
4. Plugin: `project/.claude-plugin/plugin.json`
	1. `project/.claude-plugins/skills`

동작 매커니즘, 우선 순위에 따라 skill 이름이 같은 경우 개인의 skill 이 무시될 수 있다.
- 보다 더 명확하게 구분 된 skill 을 만들어서 사용하는게 쉬운 해결책

### Plugin Skills Not Appearing
- clear the cache, restart Claude Code.
- `skills-ref` 명령어를 활용할 시점이다!

### Runtime Errors
- 의존성 부족: skill description 에 의존성 정보를 추가해서 Claude 가 알 수 있도록 하자.
- 권한 문제: skill 이 참조하는 script 에 `chmod +x` 와 같은 명령어를 추가하면 쉽게 해결.
