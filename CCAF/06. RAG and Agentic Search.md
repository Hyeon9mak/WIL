## RAG 가 왜 필요한가
### 대용량 문서의 문제점
결국 초대용량의 문서들은 사용자가 원하는 모든 요구사항을 해소해줄 수 있는 정보를 갖고 있지만, 이를 통째로 Claude 에게 던져서 질문에 사용할 순 없다. prompt 에 포함할 수 있는 context 양에는 제한이 존재한다. prompt 에 다 때려 박으면 결국..

- Claude 는 너무 큰 context (prompt) 에서 효과가 떨어진다.
- context 가 너무 크면 결국 처리 비용과 시간이 늘어난다.

### 문서를 Chunk 분할
문서를 미연에 더 작은 Chunk 단위로 분할하고, 사용자의 질문에 맞춰 가장 연관성이 높은 Chunk 를 찾아 prompt 에 포함시킨다.

- Claude 가 연관성이 높은 콘텐츠에 빠르게 집중한다.
- 사실상 문서 자체 크기의 제한이 사라진다.
- 여러 문서를 섞어서 병행할 수 있고
- 비용이 적게 들고 빨라진다.

### RAG 의 문제점
당연히 장점만 있는 것은 아니다. 사전 작업이 많이 필요하다.

- 문서를 Chunk 단위로 분할하는 전처리 프로세스
	- 어떤 방식이 최선인지도 고민이 많아야 함.
- Chunk 를 찾아내는 검색 매커니즘
- Chunk 에 정답이 없는 (Chunk 를 잘못 찾는) 일이 생길 수 있음

RAG 는 유지보수, 관리가 굉장히 어렵고 복잡한 방식이다. 결국 RAG 를 도입 했을 때 이점이 애플리케이션 전체의 복잡성을 야기하는 것보다 이득인지 고민을 많이 해봐야 한다.

## Text Chunking
결국 문서를 얼마나 올바른 기준으로 Chunking 해두는지가 시작이라고 볼 수 있다. 여기서 Chunk 의 품질이 나쁠 경우 Claude 가 잘못된 답변을 제공할 수 밖에 없다.

### Size-Based Chunking
- 가장 가볍고 간단한 접근 방식.
- 문자열을 동일한 길이로 나눈다.
- 그러나 Chunk 가 문맥을 잃어버린다는 것이 가장 크리티컬한 문제점이다.
	- 이를 보완하기 위해 Chunk 끼리 겹치는 문자열을 더해서 저장한다.
	  인접한 문자열을 포함시켜서 문맥을 유지하는 방식

```python
def chunk_by_char(text, chunk_size=150, chunk_overlap=20):
	chunks = [] 
	start_idx = 0 
	
	while start_idx < len(text): 
		end_idx = min(start_idx + chunk_size, len(text)) 
		chunk_text = text[start_idx:end_idx] 
		chunks.append(chunk_text) 
		
		start_idx = ( 
			end_idx - chunk_overlap if end_idx < len(text) else len(text)
		)
		
	return chunks
```

### Structure-Based Chunking
- 선택할 수 있다면 가장 좋은 최고의 방식
- 구조 기반 청킹은 문서의 자연적인 구조(헤더, 단락, 섹션)에 따라 분할
- 각 Chunk 가 완전한 섹션을 나타내기 때문에 가장 깔끔하고 의미있는 Chunk 를 제공
	- 원장 데이터가 완벽하다는 보장이 있어야 한다.
	- 당연히 현실적으로 많은 문서는 명확한 구조적 마커가 없거나 일반 텍스트나 PDF 다.

```python
def chunk_by_section(document_text):
	pattern = r"\n## " 
	return re.split(pattern, document_text)
```

### Semantic-Based Chunking
- 문서가 비정규화 상태일 때 가장 정교한 접근 방식
- 텍스트를 문장으로 나누고, 자연어 처리를 통해 문장들이 얼마나 관련되어 있는지를 판단.
- 계산 비용이 많이 들지만, 가장 높은 품질의 Chunk 를 생성한다.

### Sentence-Based Chunking
- Semantic 방식을 이용하기 힘들다면 그 중간 지점으로 채택할 수 있는 방식
- 정규표현식을 사용해서 텍스트를 개별 문장으로 나누고, 선택적으로 겹치게 해서 Chunk 를 나눈다.

```python
def chunk_by_sentence(text, max_sentences_per_chunk=5, overlap_sentences=1): 
	sentences = re.split(r"(?<=[.!?])\s+", text) 
	
	chunks = [] 
	start_idx = 0 
	
	while start_idx < len(sentences): 
		end_idx = min(start_idx + max_sentences_per_chunk, len(sentences)) 
		current_chunk = sentences[start_idx:end_idx] 
		chunks.append(" ".join(current_chunk)) 
		
		start_idx += max_sentences_per_chunk - overlap_sentences 
		
		if start_idx < 0: 
			start_idx = 0 
	
	return chunks
```


## Text Embeddings
문서 Chunking 이 잘 되어 있다면, 다음 단계는 사용자 질문과 관련성이 높은 Chunk 를 잡아내는 것이다. 이는 기본적으로 검색 엔진과 유사한 문제를 겪는다.

### Semantic Search
가장 일반적인 방법은 의미론(semantic) 검색이다. text embedding 을 활용해서 각 사용자의 질문과 텍스트 청크의 의미와 맥락을 교차시킨다.

<img width="1280" height="720" alt="Image" src="https://github.com/user-attachments/assets/0804ddd3-14aa-4672-82ce-0207c937e04e" />

`-1` ~ `+1` 사이의 숫자가 서로 얼마나 유사한지 비교하면서 잡아내는 것.

### 어떻게 Embedding 을 제공하는가?
Anthropic 에서 Embedding API 를 제공하진 않는다. 여러 구현체들이 있으니 그것들을 이용해서 Embedding 을 수행하면 된다.

얻어낸 embedding 결과물을 vector DB 에 저장한다. vector DB 는 embedding 과 같은 긴 숫자 목록을 저장, 비교, 검색하도록 특화된 DB 다.

## 어떻게 동작하는가?
- 사용자 질문이 전달되면, 해당 질문을 동일한 model 을 사용하여 embedding 화 시킨다.
- embedding 된 질문과 vector DB 에 적재된 embedding 값을 비교해서 가장 유사한 값을 조회해온다.
	- 유사성 동작 원리는 Cosine 유사성
- 마지막으로, 찾아낸 텍스트 Chunk 를 prompt 와 결합시켜 Claude 에게 보낸 후 응답을 받는다.

```

```

## BM25 Lexical Search
INC-2023-Q4-011 와 같은 특정 Key 를 기준으로 문서에서 Semantic 검색을 수행한다고 가정해보자. Semantic 검색은 일반적으로 맥락과 의미를 이해하는데 뛰어나지만, 이런 특정한 고유명사, ID 등에는 취약한 모습을 보인다.

### BM25 작동 원리
BM25 (Best Match 25) 는 lexical 검색 중 가장 인기가 좋다.

<img width="1280" height="720" alt="Image" src="https://github.com/user-attachments/assets/12f4be33-821b-4bcb-952e-d9085c0dcdb8" />

1. 쿼리 tokenization. 사용자의 질문을 개별 용어로 나눈다.
   "a INC-2023-Q4-011"은 ["a", "INC-2023-Q4-011"]이 된다.
2. 용어 빈도 계산. 각 용어가 모든 문서에 걸쳐 얼마나 자주 나타나는지 확인.
   "a"와 같은 일반적인 단어는 5번 나타날 수 있고, "INC-2023-Q4-011"과 같은 특정 용어는 1번만 나타날 수 있다.
3. 용어를 중요도로 가중치 부여. 자주 나타나지 않는 용어가 더 높은 중요도 점수를 받음.
   "a"라는 단어는 흔하기 때문에 낮은 중요도를 받지만, "INC-2023-Q4-011"은 드물기 때문에 높은 중요도를 받음.
4. 최상의 일치 항목 찾기. 더 높은 가중치를 부여받은 용어를 더 많이 포함하는 문서를 반환한다.

이를 코드로 구현하면 다음과 같다.

```python
chunks = chunk_by_section(text)

store = BM25Index()
for chunk in chunks: 
	store.add_document({"content": chunk})

results = store.search("What happened with INC-2023-Q4-011?", 3) 

for doc, distance in results: 
	print(distance, "\n", doc["content"][:200], "\n----\n")
```
