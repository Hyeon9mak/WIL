## When to Use workflows vs agents

- Claude 가 문제 해결을 위해 해야할 일을 정확하게 나타내거나 표현할 수 있을 때. 즉, 명확한 상황에서는 workflow 가 훨씬 효율적이다.
- 반대로 어떤 작업 또는 어떤 파라미터를 전달할지 모호한 경우 agent 를 활용한다.

## 유용한 workflow pattern: Evaluator-optimizer

- Producer: 입력을 받아 출력 생성

디자인 패턴과 같이 이러한 유명 패턴들을 통해서 패턴화된 문제에 대해 빠르게 인지하고 접근해서 해결해볼 수 있다.

## parallelization workflows

복잡한 작업을 관리 가능하고 집중된 부분들로 divide & conquer 할 수 있도록 도와주는 pattern 이 바로 parallelization workflows.

예를 들어, 이미지를 Claude 에 보내고 금속, 폴리머, 세라믹, 복합재료, 엘라스토머 또는 목재 중에서 선택하도록 하는 간단한 prompt 를 떠올려보자. 각 재료 유형에 대한 구체적인 기준이 없으면 신뢰도가 떨어질 것이다.

<img width="1280" height="720" alt="Image" src="https://github.com/user-attachments/assets/65b287e5-9edb-4464-86a7-d5b033efa8dc" />

그렇다고 기준들을 모두 prompt 하나에 거대하게 몰아놓을 순 없다.


<img width="1280" height="720" alt="Image" src="https://github.com/user-attachments/assets/2e1acb9a-d6f5-4699-996d-9abf3bb4e42c" />

각각의 조건들을 if 분기문처럼 나눠서 병렬로 요청할 수 있다. 각 요청은 전문화된 단일 기준을 활용해서 빠르게 평가하는데 집중한다.

1. Claude 에 동일한 이미지를 동시에 여러 번 전송
2. 각 요청에는 하나의 재료에 대한 전문화된 기준 포함 (금속 기준, 폴리머 기준, 세라믹 기준 등).
3. Claude 는 각 재료에 대해 부품의 적합성을 독립적으로 평가
4. 모든 분석 결과를 수집하여 최종 집계 단계에 입력

<img width="1280" height="720" alt="Image" src="https://github.com/user-attachments/assets/29248214-2b83-4813-996d-a857bd879257" />

분할정복 방식으로 인해, Claude 가 여러 사고 사이에 균형을 잡으려 하기보다 하나의 특정 측면에만 집중할 수 있다. 더욱 철저하고 정확한 분석으로 이어진다.
개발자 입장에서도 최적화가 훨씬 더 쉬워진다. workflow 각각에 대해서만 집중하면 되기 때문에.
새로운 재료를 추가하는 것 또한 편하고 쉽다. 확장성까지 뛰어난 방식.

## Chaining workflows

단순해보이겠지만, 가장 유용한 패턴 중 하나. 특히 Claude 가 일관되게 처리하기 어려운 복잡한 작업이나 긴 prompt 를 다룰 때 특히 가치가 커진다.

<img width="1280" height="720" alt="Image" src="https://github.com/user-attachments/assets/51e7936f-b93b-4c24-98bb-43e304b76637" />

결국 크고 복잡한 작업을 작게 분할한 뒤, 순차적으로 시키는 것이다. parallelization workflows pattern 과 마찬가지로 Claude 가 한 번에 하나의 구체적인 작업에 집중하는 것에서 차이가 생긴다. 개발자 역시도 최적화가 쉬워지는 것은 덤이다.

workflow 가 지나치게 크고 길어지면 Claude 는 결국 일부 규칙을 위반하거나 틀리기 마련이다.

이 방식은 Server 개발 뿐만 아니라, 대화형으로 Claude 를 이용할 때도 통하는 규칙.

## Routing workflows

만능 prompt 라는 건 세상에 존재하지 않는다. 결국 들어오는 요청을 분류하고 특화된 처리 파이프라인으로 라우팅하는 것이 가장 이롭다.

가령, 프로그래밍 주제는 명확한 설명과 정의가 있는 교육 콘텐츠를 필요로 한다.
서핑 주제는 흥분감과 시각적 매력을 강조하는 엔터테인먼트 중심의 스크립트가 더 잘 작동한다.
하나의 일반적인 prompt 로는 둘을 모두 효과적으로 처리할 수 없다.

### 접근 방식
첫 번째는 콘텐츠 유형을 분류하는 등의 카테고리화. 애플리케이션이 구분지어야 할 영역이 무엇인지 명확히 이해하고 카테고라이징을 준비한다. 그 후 다음 절차를 거쳐서 라우팅을 시킬 수 있다.

1. **Categorization**: 사용자의 주제를 Claude 에게 보내 미리 정의한 장르 중 하나로 분류하도록 요청
2. **Specialized Processing**: 카테고리 결과를 사용하여 적절한 prompt 템플릿을 선택하고 콘텐츠 생성

```python
Categorize the topic of a video into one of the listed categories:
<topic>Python functions</topic>

<categories>
- Educational
- Entertainment  
- Comedy
- Personal vlog
- Reviews
- Storytelling
</categories>
```



## Agents and tools

workflow 는 작업을 완료하기 위해 필요한 정확한 단계를 알고 있을 때 완벽하지만, agent 는 어떤 단계를 취해야 할지 확실하지 않을 때 빛을 발한다. 엄격한 순서를 정의하는 대신, Claude 에 목표와 Tool 만 제공한 후 이를 결합하여 스스로 해결하도록 요구한다.

광범위한 문제 해결에 좋지만, 신뢰성과 비용 측면에서 분명 트레이드오프를 갖게 된다.

### How tool make the Agent

각각의 간단한 tool 들을 Claude 가 어떻게 상황별로 다르게 조합해서 사용하는지 살펴보자.

<img width="1280" height="720" alt="Image" src="https://github.com/user-attachments/assets/962abdc2-5c34-48b0-862d-975e70834c55" />

이와 같이 유동적인 Tool 활용을 위해선, Tool 이 충분히 추상적이어야한다.
각 영역별로 스페셜한 tool 을 구성하는 게 중요하지만, 그 강도가 지나칠 경우 유연하게 광범위한 상황을 처리할 수 없게 된다.

> 이 시점에서, workflow 와의 차이점은. agent 에게 기대한다는 것 자체부터 이미 모호한 상황을 가정하고 있기 때문에. 그렇지 않고 명확하게 스페셜한 tool 을 이용한다면 그것부터가 agent 가 아닌 workflow 로 충분히 풀어낼 수 있는 상황이라는 의미 같음.

가령, Claude Code 는 "refactor code", "run tests" 같은 tool 은 없지만, "bash", "read", "write", "glob", "grep" 등과 같은 Tool 들을 갖고 있다.

결국, 추상적이고 유연한 Tool 이 Tool 끼리의 결합을 만들고 더 광범위한 문제를 해결하는데 도움을 준다.


## Environment inspection

Claude 는 컴퓨터 내부 상황이나 프로젝트 상황을 읽을 방법이 없기 때문에, 계속해서 사용자에게 환경 검사, 환경 주입을 요구한다. Claude 의 관점에서 버튼을 클릭하면 새로운 페이지로 이동하거나, 메뉴가 열리거나, 수많은 변화를 유발할 수 있는데, 자신의 행동이 성공했는지 또는 환경의 새로운 상태가 어떻게 보이는지 이해할 방법이 없다.

### Reading before Writing
동일한 원칙은 당연히 파일 작업에도 적용된다. Claude 가 파일을 수정하기 전에 현재 내용을 이해해야 한다.
web_search, bash 등 Tool 과, System prompt 를 조합하여 작업 전 미리 상황을 인지하도록 지침을 내려주면 좋다.

## 총 정리

일반적인 권장사항은 가능한 한 workflow 구현에 집중하고, 정말 필요한 경우에만 agent 를 사용하는 것.

workflow 는 대부분의 프로덕션 애플리케이션이 필요로 하는 안정성과 예측 가능성을 제공하며,
agent 는 정확한 요구사항을 미리 정할 수 없는 시나리오에서 유연성을 제공한다.
