## Coding Assistant

Coding assistant 는 단순히 코드를 작성하는 도구가 아니다. Language model 을 활용해서 복잡한 프로그래밍 작업을 해결하는 정교한 시스템. Assistant 가 내부에서 어떻게 동작하는지 이해하는게 중요하다.

<img width="1920" height="1080" alt="Image" src="https://github.com/user-attachments/assets/8b59521f-ff9b-436f-b448-325bb6d5071f" />

1. Gather context: context 수집 후 오류가 무엇을 나타내는지, 코드 어느 부분에 영향이 있는지 등 이해하기
2. Formulate a plan: 리팩터링 및 테스트로 수정사항을 확인하는 등 방법 결정
3. Take action: Plan 에 따라 실제 조치

Formulate a plan 과정은 model 내부에서 동작을 수행하지만, Gather context 와 Take action 은 외부 세계와 상호작용을 수행한다는 점에 주목하자. (파일 읽기, 명령 실행 등) 이 과정에서 자연스럽게 Tool 을 요구한다.

### How Tool use works

<img width="1920" height="1080" alt="Image" src="https://github.com/user-attachments/assets/e9ca57be-9acb-4755-9b48-b76af9a62439" />

1. 내가 "main.go 파일에 어떤 코드가 작성되어 있나요?" 라고 물으면,
2. coding assistant 가 Tool 사용 요청 지침을 추가한다.
3. 그리곤 language model 이 응답한다: "ReadFile: main.go"
4. coding assistant 가 실제 파일을 읽고 그 내용을 model 에 다시 보낸다.
5. language model 은 파일 내용을 기반으로 최종 답변을 제공한다.

**Claude 는 특히나 다른 model 들에 비해 Tool 활용에 강하다.**

- **Tackles harder tasks**: Claude 는 복잡한 작업을 처리하기 위해 다양한 tool 을 조합하여 사용할 수 있으며, 이전에 본 적이 없는 tool 도 사용한다.
- **Extensible platform**: Claude Code 에 새로운 tool 을 쉽게 추가할 수 있으며, Claude 는 workflow 가 변함에 따라 이러한 tool 들을 사용하도록 자동으로 적응한다.
- **Better security**: 더 나은 보안 - Claude Code 는 인덱싱을 요구하지 않고 코드베이스를 탐색할 수 있으므로, 전체 코드베이스를 외부 서버로 전송하지 않아도 됨.

## Context 추가하기

당연히, 불필요한 context 가 많아지면 Claude 의 추론 능력이 떨어지기 마련이다. 현재 요구사항에 적합한 배경지식만 습득하도록 context 를 제한해주는 것이 중요하다.

### /init

Claude Code 프로젝트에서 작업을 시작할 땐 가장 먼저 `/init` command 를 입력한다. Claude 가 전체 코드베이스를 스캔하고 프로젝트의 구조, 아키텍처 등을 이해한다. 그 후 `CLAUDE.md` 파일을 생성해서 context 를 관리한다.

그 외에도 Claude Code 내에서 `/memory` 명령어를 이용해서 파일을 직접 열 수도 있다. Claude 는 모든 대화 시작 시 이 파일을 읽으므로, 바로 다음 message 부터 변경사항이 반영된다.
(context 를 날리고 싶다면 `/clear`)

`CLAUDE.md` 는 다음과 같은 계층 구조로 나눠 만들어서 활용할 수 있다.

- **Project**: 프로젝트에서 작업하는 모든 엔지니어 간에 공유됨
	- `CLAUDE.md`
- **Local**: git에 체크인되지 않은 개인 노트
	- `CLAUDE.local.md`
- **User**: 모든 프로젝트에 걸쳐 사용됨
	- `~/.claude/CLAUDE.md`


### context 주입

Claude 에게 무언가를 요청하기 전에 정확한 context (코드베이스 파일 등) 를 주입하자. 그럼 기존 코딩 패턴과 활용할 수 있는 기존 기능 예시들을 확보할 수 있다.

Claude 가 특정 파일을 참고하도록 해야 할 때는 `@` 기호 뒤에 파일 경로를 사용.
이 방식은 `CLAUDE.md` 문서 내에서도 유효하다. 여전히 파일 참조를 같은 방식으로 시켜둘 수 있다.

저장소에 이미 다른 tool 용 AGENTS.md 파일이 있는 경우에도 유용하다!
`CLAUDE.md` 의 첫 번째 줄에 `@AGENTS.md` 를 추가하면 Claude 가 먼저 해당 파일의 내용을 로드한다.
그런 다음 import 아래에 Claude 특화 지시사항을 추가할 수 있다.

Claude 와 소통하는 가장 효과적인 방법 중 하나는 스크린샷을 첨부하는 것. 이미지 첨부를 적극 이용하자.

### 솔루션 계획 세우기 지시
바로 구현으로 넘어가기 보다, Plan mode 를 활용해서 구체적인 계획을 세우도록 유도하자.

### 솔루션 구현
구체적이고 견고한 계획이 세워지면 그 후부터 구현을 요청하자.


## Planning & Effort

### Planning mode
코드베이스 전체에 걸친 광범위한 조사가 필요한 경우, Plan mode 를 이용하면 좋다.
`/plan` 또는 `Shift + Tab` 으로 활성화가 가능하다. Plan mode 에 진입하면...

- 프로젝트의 더 많은 파일을 읽고, 상세한 구현 계획을 작성함
- 정확히 어떤 작업을 수행할 것인지 미리 보여주고, 승인을 기다림

여기서 팁은, 계획을 검토할 때 `Ctrl+G` 를 눌러 편집이 가능하다. 편집이 완료되면 Claude 는 최종 계획을 읽게 된다.

### Effort mode

Claude 는 기본적으로 사용자 답변 이전에 추론을 진행한다. `Ctrl + O` 단축키를 입력하면 이 과정을 볼 수 있다.

`/effort` 명령어를 통해 현재 수준을 확인하고 조정할 수 있다.
low 는 빠르고 저렴하며, max 는 느리고 비싸지만 확실하다. 기본 값은 model 과 요금제에 따라 다르다.

**단발성으로 깊은 추론을 요구하고 싶은 경우 `ultrathink` 라는 키워드를 이용하자.**

### 각각 언제 사용?
광범위한 이해가 필요한 작업, 여러 절차에 따른 작업과 영향 범위가 큼 --> Planning mode
복잡한 논리 문제, 어려운 문제 디버깅, 알고리즘 --> Effort higher


## Control Context

### ESC 로 Claude 중단하기

Claude 가 잘못된 방향으로 가거나 한 번에 너무 많은 것을 시도할 때가 있다. ESC 를 눌러 바로 중단하고 방향을 다잡으면 된다.

만약 ESC 로 중단하는 상황이 반복된다면? 해당 내용과 올바른 해결책을 `CLAUDE.md (/memory)` 에 업데이트하자. 모든 대화에서 해당 내용을 참조하기 때문에 동일한 오류 발생을 막게 된다.

### commands

대화가 길어지면 불필요한 context 가 중첩돼서 Claude 가 오류를 범하고 디버깅이 길어진다.
ESC 2회 입력 혹은 `/rewind` 를 입력해서 대화를 되감을 수 있다.

- `/rewind`: context 되돌리기
- `/compact`: context 압축
- `/clear`: 새로운 context 로 실행. 이전 context 를 완전히 지우는 건 아님.
- `/resume`: context 탐색 후 실행. `/clear` 로 날린 context 도 찾을 수 있다.


## Custom commands

1. `.claude` 디렉토리 하위에 `commands` 디렉토리를 만든다.
2. 원하는 명령어 이름으로 마크다운 파일을 생성한다. (ex. `audit.md`)
3. 파일명이 명령어가 되어 실제로 `/audit` 을 통해 호출이 가능해진다.

파일 내부에 `$ARGUMENTS` 키워드를 활용해서 파라미터를 주입 받을 수도 있다.

```md
Write comprehensive tests for: $ARGUMENTS

Testing conventions:
* Use Vitest with React Testing Library
* Place test files in a __tests__ directory in the same folder as the source file
* Name test files as [filename].test.ts(x)
* Use @/ prefix for imports

Coverage:
* Test happy paths
* Test edge cases
* Test error states
```

그럼 다음과 같이 실행해볼 수 있다.

```
/write_tests the use-auth.ts file in the hooks directory
```


## MCP Server with Claude code

MCP (Model Context Protocol) Server 를 추가하여 Claude Code 의 기능을 확장할 수 있다.
이 서버들은 원격 혹은 개인 머신에서 실행하도록 두고, Claude Code 를 통해 활용하게 된다.

MCP Server 내부 Tool 을 처음 사용할 때, Claude 는 매번 권한을 묻는다.
이러한 권한 요청 message 가 번거로우면 설정을 편집하여 서버를 미리 승인할 수 있다.
`.claude/settings.local.json` 파일을 열고 allow 배열에 Server 정보를 추가하면 된다.

```json
{
  "permissions": {
    "allow": ["mcp__playwright"],
    "deny": []
  }
}
```


## GitHub integration

Claude Code 는 GitHub Actions 내에서 Claude 가 실행되도록 하는 공식 기능을 지원한다.
Issue 및 Pull Request 에 대한 멘션 지원, 그리고 자동 Pull request review 가 대표적이다.

시작하려면 Claude 에서 `/install-github-app` 을 실행하자.
자동으로 프로젝트 내부에 `.github/workflows` 하위에 workflow 파일을 생성한다.

### Default GitHub Actions

- mention action
	- `@claude` 키워드를 사용해서 issue, Pull Request 에서 Claude 를 부를 수 있다.
	- 요청을 분석하고 작업 계획을 수립
	- 코드베이스에 대한 전체 접근 권한으로 작업 실행
	- issue 또는 PR 에서 바로 응답
- pull request action
	- PR 이 생성될 때마다 자동으로 변경 사항 검토
	- 영향도 분석
	- 상세 보고서 게시


## Hooks!
Hooks 를 사용하면 Claude 가 특정 작업을 실행하기 전이나 후에 명령을 실행할 수 있다. 이벤트 처리 방식이라고 생각하면 된다.

<img width="1920" height="1080" alt="Image" src="https://github.com/user-attachments/assets/5b9754e0-ad3c-4332-9864-6216bd4531a7" />

여러가지 Hook 들이 많은데, 일단 대표적으로는

- **PreToolUse hooks**: Tool 이 호출되기 전에 실행
- **PostToolUse hooks**: Tool 이 호출된 후에 실행

### 어떻게 구성하는가?

Hook 은 Claude 설정 파일을 통해 정의된다.

- **Global** - `~/.claude/settings.json` (affects all projects)  
- **Project** - `.claude/settings.json` (shared with team)  
- **Project (not committed)** - `.claude/settings.local.json` (personal settings)  

혹은 Claude Code 내에서 `/hooks` 명령을 사용해서 정의할 수도 있다.

<img width="1920" height="1080" alt="Image" src="https://github.com/user-attachments/assets/229b4252-57fd-45e6-93b9-b895273aa410" />

예시를 간략히 보자.

```json
"PreToolUse": [
  {
    "matcher": "Read",
    "hooks": [
      {
        "type": "command",
        "command": "node /home/hooks/read_hook.js"
      }
    ]
  }
]
```

'Read' Tool 이 실행되기 전에 이 구성이 지정된 명령을 실행한다. 
명령은 Claude 가 수행하려고 하는 Tool 호출에 대한 세부 정보를 받으며, 다음과 같은 동작을 명세할 수 있다.

- 작업이 정상적으로 진행 되도록 허용
- Tool 호출을 차단하고 Claude 에 오류 message 전송

```json
"PostToolUse": [
  {
    "matcher": "Write|Edit",
    "hooks": [
      {
        "type": "command", 
        "command": "node /home/hooks/edit_hook.js"
      }
    ]
  }
]
```

Tool 호출이 이미 발생했으므로 PostToolUse hook 은 작업을 차단할 수 없다.
그러나 다음을 수행할 수 있다.

- 후속 작업 실행 (방금 편집된 파일을 포맷하는 등)
- Tool 사용에 대한 추가 피드백을 Claude 에 제공

그 외에도 코드 포맷팅, 테스팅, 접근 제어, 로깅 등 여러가지 반복 규칙 작업을 Hook 으로 명세할 수 있다.


## Building a Hook

1. 어느 쪽(전/후) 이벤트에 Hook 을 걸지 결정
2. (Hook 에 따라)정확히 어떤 유형에 트리깅 되도록 할지 결정
3. 트리깅 기준이 될 명령을 작성한다.
4. (필요한 경우) 명령이 Claude 에 피드백을 제공해야 한다.
   명령의 종료 코드는 Claude 에게 작업 허용 또는 차단 여부를 알려주어야 한다.

JSON 을 읽고 어떤 이름의 Tool 에 대해 동작할지 결정

<img width="1920" height="1080" alt="Image" src="https://github.com/user-attachments/assets/79401c8c-cdc3-47fb-b0aa-6f31838981a9" />

일반적인 사용 사례는 Claude 가 `.env` 파일과 같은 민감한 파일을 읽지 못하도록 방지하는 것.
Read 와 Grep Tool 모두 파일 내용에 접근할 수 있으므로, 두 tool 유형을 모니터링하고 제한된 파일 경로에 접근하려고 하는지 핸들링 할 수 있다.

```javascript
// .claude/settings.local.json

{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Read",
        "hooks": [
          { "type": "command", "command": "node $PWD/hooks/read_hook.js" }
        ]
      }
    ]
  }
}
```

그리고 `hooks/read_hook.js` 이름으로 hook 구현체를 만들어보자.

```javascript
process.stdin.setEncoding("utf8");
let input = "";
process.stdin.on("data", (d) => (input += d));
process.stdin.on("end", () => {
  const toolArgs = JSON.parse(input);
  const readPath = toolArgs.tool_input?.file_path || "";
  if (readPath.includes(".env")) {
    console.error("You cannot read the .env file");
    process.exit(2);
  }
  process.exit(0);
});
```

stdin 에서 Tool 정보를 JSON 형태로 읽고, tool_input.file_path 를 확인 후 차단할 것이다.

**권장사항은 무얼 이용하든, 상대경로가 아닌 절대 경로를 이용하는 것.**


## 유용한 Hooks!

### compile hook
Claude 가 코드베이스를 수정한 후 모든 호출처를 업데이트 하지 않는 경우가 있다. 이에 대해 미연에 검사를 진행하도록 하는 hook 을 준비하는 것. 당연히 `post-tool-use` hook 이 되겠다.

### Query duplication prevention hook
기존에 동일한 쿼리가 있는지 사전 검토 후 반영을 고민하는 것.
Claude 가 persistence 디렉토리 하위 파일을 수정할 때 트리깅 되도록 유도하면 된다.

### 고려 사항

Hook 은 어쨌거나 추가 리소스를 사용하는 작업이다. 그 비용 대비 효용성이 얼마나 큰지 항상 고민하고 도입할 것.

## Tool 외 hook 기준

- `Notification`
	- Claude Code 가 알림을 전송할 때
	- Claude 가 tool 을 사용할 권한이 필요하거나 Claude Code 가 60초 동안 유휴 상태였을 때 발생
- `Stop`
	- Claude Code 가 응답을 마쳤을 때
- `SubagentStop`
    - subagent (UI 에서 "작업" 으로 표시됨) 가 완료되었을 때
- `PreCompact`
	- 수동 또는 자동 압축 작업이 발생하기 전
- `UserPromptSubmit`
    - 사용자가 prompt 를 제출할 때, Claude 가 처리하기 전
- `SessionStart`
    - 세션을 시작하거나 재개할 때
- `SessionEnd`
    - 세션이 종료될 때


## Agent SDK

Agent SDK 를 사용하면 애플리케이션과 스크립트에서 Claude Code 를 프로그래밍 방식으로 실행할 수 있다.
TypeScript 와 Python 에서 사용 가능.
CLI 가 사용하는 것과 동일한 agent 루프 (파일 읽기, 편집, tool 사용) 를 제어 하에 제공.

### 설치

```bash
$ mkdir sdk-demo
$ cd sdk-demo
$ npm init -y
$ npm install @anthropic-ai/claude-agent-sdk
```

### 예시

```javascript
import { query } from "@anthropic-ai/claude-agent-sdk";

const prompt = "List the files in the current directory";

for await (const message of query({ prompt })) {
  console.log(JSON.stringify(message, null, 2));
}
```


```bash
node index.mjs
```


JSON message 의 스트림이 표시된다.
CLI 에서 보게 되는 것과 동일한 대화 이벤트이며, 여기에는 tool 호출, tool 결과, Claude 의 텍스트가 포함.
