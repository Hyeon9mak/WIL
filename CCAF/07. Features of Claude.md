## 확장 사고 모드 (Extended Thinking)

> 일부 기능과는 호환되지 않는다. 자세한 제약사항은 아래 링크 참고.
> [**https://docs.anthropic.com/en/docs/build-with-claude/extended-thinking#feature-compatibility**](https://docs.anthropic.com/en/docs/build-with-claude/extended-thinking#feature-compatibility)

**확장 사고 모드는 prompting 이 충분하지 않을 때를 지원하는 도구다.**

확장 사고 모드는 Claude 의 고급 추론 기능. model 이 최종 응답을 생성하기 전 해결시간을 더 많이 제공한다. 답변으로 이어지는 추론 과정을 사용자에게 그대로 노출하기 때문에 투명성이 높아지고 더 높은 품질을 기대할 수 있게 된다.

### 동작 원리
확장 사고가 활성화되면, Claude 의 응답은 단순한 Text Block 에서 **Text Block 과 Thinking Block** 으로 구성된 구조화된 응답으로 변경된다.

<img width="1280" height="720" alt="Image" src="https://github.com/user-attachments/assets/fdf1423a-ca9a-4acb-ba31-0d5275404a02" />

즉 추론 과정(Thinking Block)과 최종 답변(Text Block)을 모두 얻을 수 있는 얘기.

<img width="391" height="290" alt="Image" src="https://github.com/user-attachments/assets/891ff01a-9ec2-44b5-92f4-05bd2757e9c8" />


Thinking Block 에는 `signature` 필드를 통해 암호화 token 을 포함하고 있다. Claude 의 추론 과정을 개발자가 임의로 조작하는 것을 방지하기 위한 수단.

- 복잡한 작업에 대한 향상된 추론 능력
- 어려운 문제 정확도 증가
- 사고 과정에 대한 투명성
- 하지만 더 높은 비용
- 하지만 더 증가된 지연 시간
- 코드 처리 복잡도 증가

### 사용법

코드에서 사용을 위해선 채팅 함수에 매개변수를 2개 추가한다.

```python
def chat(
	messages, 
	system=None, 
	temperature=1.0,
	stop_sequences=[], 
	tools=None, 
	thinking=False, 
	thinking_budget=1024
):
```

`thinking` 을 통해 활성화 시킬 수 있으며, `thinking_budget` 으로 추론에 활용할 최대 token 을 설정한다. 최소값은 1024 이고, `max_tokens` 를 제한할 땐 당연히 `thinking_budget` 보다 `max_tokens` 값이 더 커야한다.

```python
if thinking:
	params["thinking"] = { 
		"type": "enabled", 
		"budget": thinking_budget
	}
```

### 확장 사고 모드를 적용할 시기
간단하다. prompt 평가를 준비해두고, prompt 를 최대한으로 최적화 했음에도 문제 해결을 하지 못하면 그 때 활성화를 고려하자. **확장 사고 모드는 prompting 이 충분하지 않을 때를 지원하는 도구다.**


## 이미지 지원(Image support)

Claude vision 기능을 사용하면 질의문에 이미지를 포함시키고 다양한 방식으로 Claude 가 분석하도록 요청할 수 있다.

- 단일 요청 내 최대 100개의 이미지
- 이미지당 최대 5MB 크기 제한
- 이미지 1개 전송 시: 최대 높이/너비 8000px
- 이미지 여러 개 전송 시: 최대 높이/너비 2000px
- 이미지는 base64 인코딩 또는 이미지 URL 로 포함할 수 있다.
- 각 이미지는 크기에 따라 token 으로 계산 `(width px * height px) / 750` 된다.

### 사용법

```python
with open("image.png", "rb") as f:
	image_bytes = base64.standard_b64encode(f.read()).decode("utf-8") 
	
add_user_message(messages, [ 
	# Image Block 
	{ 
		"type": "image", 
		"source": { 
			"type": "base64", 
			"media_type": "image/png", 
			"data": image_bytes, 
		} 
	}, 
	# Text Block 
	{ 
		"type": "text", 
		"text": "What do you see in this image?" 
	} 
])
```

## PDF 지원(PDF support)

이미지 처리와 유사하지만 코드를 구성하는 방식에서 몇 가지 주요 차이점이 존재한다.

```python
with open("image.png", "rb") as f:
	file_bytes = base64.standard_b64encode(f.read()).decode("utf-8")
	
messages = [] 

add_user_message( 
	messages, 
	[ 
		{ 
			"type": "document", 
			"source": { 
				"type": "base64", 
				"media_type": "application/pdf", 
				"data": file_bytes, 
			}, 
		}, 
		{"type": "text", "text": "Summarize the document in one sentence"}, 
	], 
) 

chat(messages)
```

- 확장자명 변경 주의
- 타입을 `image` 가 아닌 `document` 로 변경
- 미디어 타입을 `image/png` 가 아닌 `application/pdf` 로 변경

### Claude 가 PDF 에서 알아낼 수 있는 것

- 문서 전체의 텍스트
- PDF 에 포함된 이미지 및 차트
- 테이블과 그들의 데이터 관계
- 문서 구조 및 형식

Claude 는 PDF 를 예상보다 훨씬 더 잘 이해하고 분석한다.


## 인용(Citations)

Claude 가 특정 정보를 정확히 어디서 획득했는지 보여줄 수 있다면 신뢰도가 확실히 올라갈 것이다.
인용(Citations) 기능은 이 문제를 완벽히 해결해준다.

### 사용법

```python
{
	"type": "document", 
	"source": { 
		"type": "base64", 
		"media_type": "application/pdf", 
		"data": file_bytes, 
	}, 
	"title": "earth.pdf", 
	"citations": { "enabled": True }
}
```

인용이 활성화되면 Claude 의 응답 구조가 자연스럽게 복잡해진다. Claude 의 모든 주장에 대한 인용 정보를 포함하는 구조화된 답변을 받기 때문이다.

- `cited_text`: 인용 텍스트
- `document_index`: Claude 가 참조하는 문서
- `document_title`: 문서 제목(이름)
- `start_page_number`: 인용 텍스트 시작 위치
- `end_page_number`: 인용 텍스트 종료 위치

PDF 뿐만 아니라 평문을 직접 제공하여 인용하도록 유도할 수도 있다.

```python
{
	"type": "document", 
	"source": { 
		"type": "text", 
		"media_type": "text/plain", 
		"data": article_text, 
	}, 
	"title": "earth_article", 
	"citations": { "enabled": True }
}
```


### Prompt Caching
prompt caching 은 이전 요청의 계산 작업을 재사용하여 Claude 의 응답속도를 높이고 텍스트 생성 비용을 줄이는 기능. 요청 이후 처리 작업을 버리지 않고 유사한 콘텐츠에 대해 재활용하는 것.

### 원래는 어떻게 동작할까?

<img width="1280" height="720" alt="Image" src="https://github.com/user-attachments/assets/fa21fbeb-59b8-4b58-a206-48811104f66a" />
<img width="1280" height="720" alt="Image" src="https://github.com/user-attachments/assets/99f20501-0c27-484a-a038-59e6085b4a78" />

- prompt 를 더 작은 조각으로 tokenization
- 각 token 에 대한 embedding 형성
- 주변 텍스트를 기반으로 context 보충
- 실제 출력 텍스트 생성

보통의 응답 과정에서는 이 모든 계산 작업을 폐기한다.

이어지는 질문/응답 과정이 관련이 없다면 괜찮지만, 동일한 콘텐츠를 포함하는 후속 요청 시엔 효율성이 아쉽다.
prompt caching 은 출력을 제외한 나머지 계산 작업을 저장함으로서 workflow 를 아래와 같이 변경한다.

<img width="1280" height="720" alt="Image" src="https://github.com/user-attachments/assets/5dd152cf-cc03-472b-b0ce-cbf19983bdec" />

### 이점과 한계

- 더 빠른 응답과 더 낮은 비용, 자동으로 최적화 효과
- 그러나 caching 은 1시간만 유지된다.
- 'caching' 의 뜻에 걸맞게, 반복성이 높아야 효과적이다.

### 사용법
cache 는 자동으로 활성화 되지 않는다. message block 내부 특정 지점에 **'cache breakpoint**' 를 지정해줘야한다. 그러면 breakpoint 이전의 모든 작업이 caching 된다. breakpoint 까지 내용이 동일한 경우에만 cache 가 재사용 되는 것. **정확히 동일해야한다. 단어 하나만 차이가 생겨도 cache 를 활용하지 않는다.**

breakpoint 이후 작업은 caching 없이 평소처럼 동작한다.

<img width="1280" height="720" alt="Image" src="https://github.com/user-attachments/assets/121c3a22-dc49-4633-aaa2-938314b5e832" />

cache 를 활성화 시키려면 축약형이 아닌 원문 형식으로 모든 걸 명세해야한다.
`cache_control` 필드를 통해 breakpoint 를 지정해줄 수 있다.

cache 비효율을 막기 위해 최소 Content Length 제약이 있다. **1024 tokens long**
이는 모든 message, block 의 합계이다. 개별로 1024 token 이어야하는 것은 아님. caching 동작 방식을 이해하면 자연스럽다.

message block 여러 개에 걸쳐서 breakpoint 를 걸 수도 있다. 그럼 breakpoint 가 지정된 message block 이전의 message block 들까지도 전부 cache 콘텐츠로 포함된다. 이는 특정 시점까지의 전체 context 를 cache 하고 싶은 대화에서 특히 유용하다.

- System prompts
- Tool definitions
- Tool use and tool result blocks
- Image blocks

꼭 text block 뿐만 아니라 위 다양한 타입들에서도 모두 breakpoint 설정을 통한 cache 활용이 가능하다.
System prompts 와 Tool definition 은 거의 변하지 않는 정적 내용들이니 오히려 효율이 더 좋다.

- breakpoint 는 한 번에 최대 4개까지 동시 배치가 가능하다.
- Claude 는 Tool, System prompt, messages 순서로 구성요소를 처리한다.
	- 이 순서를 기억해서 breakpoint 를 구성하면 최고의 효율을 얻을 수 있다.
	- 가령, System prompt 와 User Message 를 바꿔도 Tool cache 는 보존할 수 있게 된다.

## Prompt Caching
prompt caching 은 Claude 에 동일한 콘텐츠를 반복해서 전송할 때 API 요청을 더 빠르고 저렴하게 만드는 강력한 최적화 기능.

prompt caching 을 활성화하면 마찬가지로 1시간 동안 유지되는 cache 에 콘텐츠를 기록한다.

Large system prompts, complex tool schema 에 특히 강한 모습을 보여준다. 반복되는 message 콘텐츠도 당연하다.


### 사용법

```python
if tools:
	tools_clone = tools.copy() 
	last_tool = tools_clone[-1].copy() 
	last_tool["cache_control"] = {"type": "ephemeral"} 
	tools_clone[-1] = last_tool 
	params["tools"] = tools_clone
```

Tool schema 에도 역시 `cache_control` 을 명세해서 cache 활성화를 명시할 수 있다.

```python
if system: 
	params["system"] = [ 
		{ 
			"type": "text", 
			"text": system, 
			"cache_control": {"type": "ephemeral"} 
		} 
	]
```

System prompt cache 역시도 cache 제어를 text block 으로 구조화 해서 명세한다.

마찬가지로 cache 는 굉장히 민감해서, 글자 하나만 바뀌어도 내용이 바뀐다.

응답에서 `type: "code_execution_output"` 필드를 포함한 block 을 찾자. 그리고 다음을 이용해서 다운로드를 진행하면 된다.

```
download_file("file_id_from_response")
```
